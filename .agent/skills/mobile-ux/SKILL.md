# モバイルUX原則

モバイルアプリケーションのUX設計原則。
**Flutter、Expo、React Native共通**で適用される考え方。

---

## 基本原則

### 1. 親指ゾーン（Thumb Zone）

```
画面下部 → 操作しやすい（主要アクション）
画面中部 → 普通（コンテンツ）
画面上部 → 操作しにくい（補助的な操作）
```

**推奨:**
- 主要操作は画面下部に配置
- FAB（Floating Action Button）は親指が届く位置
- ナビゲーションバーは下部

### 2. タッチターゲット

| プラットフォーム | 最小サイズ |
|----------------|-----------|
| iOS | 44×44pt |
| Android | 48×48dp |

**推奨:**
- ボタン間に十分な余白
- タップ領域はボタン外側まで広げる

### 3. フィードバック

```
タップ → 即座に視覚的フィードバック（リップル、色変化）
処理中 → ローディングインジケータ
完了 → 成功/失敗の明示
```

## ナビゲーションパターン

### 1. ボトムナビゲーション

```
推奨: 3-5項目
用途: メイン機能への素早いアクセス
```

### 2. スタックナビゲーション

```
用途: 階層的なコンテンツ
戻る操作: スワイプ（iOS）、戻るボタン（Android）
```

### 3. ドロワーナビゲーション

```
用途: 5+項目、補助的な機能
注意: メイン機能には向かない
```

## 画面設計

### 情報密度

```
❌ 詰め込みすぎ → 視認性低下
✅ 適度な余白 → 読みやすい
```

### スクロール vs ページング

| パターン | 用途 |
|---------|------|
| 無限スクロール | フィード、リスト |
| ページング | ウィザード、オンボーディング |

### Empty State

```
データがない場合:
- 状況説明
- 次のアクション提示
- イラスト（任意）
```

## フォーム設計

### 入力フィールド

1. **ラベルは常に表示**（プレースホルダーだけはNG）
2. **適切なキーボードタイプ**（email、number、phone等）
3. **エラーは即座に表示**（送信後ではなくリアルタイム）

### バリデーション

```
✅ 良い: フィールド離脱時に即座にバリデーション
❌ 悪い: 送信ボタン押下後にまとめてエラー表示
```

## パフォーマンス体感

### 体感速度の改善

| 技法 | 説明 |
|------|------|
| Skeleton UI | データ読み込み中の骨格表示 |
| Optimistic Update | 結果を先に表示、失敗時に戻す |
| プログレッシブ表示 | 一部を先に表示 |

### 応答時間の目安

| 時間 | ユーザー体感 |
|------|------------|
| ~100ms | 即座 |
| ~1s | 遅れを感じる |
| ~3s | 待たされている |
| 3s~ | 離脱リスク |

---

## アクセシビリティ

### 必須対応

1. **スクリーンリーダー対応**: 適切なラベル設定
2. **色だけに頼らない**: 形状やテキストでも情報伝達
3. **十分なコントラスト**: WCAG基準（4.5:1以上）

---

## プラットフォーム固有

### iOS

- スワイプバック
- 慣性スクロール（バウンス）
- HIG（Human Interface Guidelines）準拠

### Android

- 戻るボタン
- Material Design準拠
- Edge-to-Edge設計
