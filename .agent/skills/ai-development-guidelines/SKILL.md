# AI駆動開発ガイドライン（普遍原則）

AI（LLM）を活用した開発で発生しやすい問題と、その対策を定義する。
**全プラットフォーム・全フレームワーク共通**で適用される原則。

---

## 🚫 禁止行動（MUST NOT）

### 1. 簡易実装への逃げ
```
❌ 禁止: エラーが困難だからと言って、簡易的な処理で完了とする
❌ 禁止: 複雑な処理をスキップして「TODO: 後で実装」で済ませる
❌ 禁止: 「時間がかかるので」を理由に機能を省略する

✅ 必須: 困難な場合はユーザーに相談し、代替案を提示する
✅ 必須: 制限事項があれば明示的に報告する
```

### 2. モック・スタブでの代替
```
❌ 禁止: 本番コードをモックで置き換えて完了とする
❌ 禁止: API未実装を理由にハードコード値で代替
❌ 禁止: ダミーデータで動作確認して「完了」とする

✅ 必須: 実際の実装を行う
✅ 必須: モックが必要な場合はテストコード内のみ
```

### 3. 無断のコード削除・変更
```
❌ 禁止: 理解できないコードを「不要」と判断して削除する
❌ 禁止: 既存機能を無断で削除・変更する
❌ 禁止: 「使われていないようだ」を理由に削除する

✅ 必須: 削除前にユーザーに確認する
✅ 必須: 削除した場合は明示的に報告する
```

### 4. エラー・警告の無視
```
❌ 禁止: コンパイルエラーを放置する
❌ 禁止: 型エラーを無視する
❌ 禁止: 警告を「問題ない」と判断して放置する

✅ 必須: 全エラー・警告を解消してから完了とする
✅ 必須: 解消できない場合は理由を説明する
```

### 5. ビルド検証の省略【重要】
```
❌ 禁止: 静的解析のみで完了とする
❌ 禁止: 「コンパイルは通るはず」で終わる
❌ 禁止: ランタイムエラーを確認せずにpushする

✅ 必須: 実際にビルド・実行して動作確認する
✅ 必須: ビルドコマンドを実行して成功を確認する
```

### 6. 諦め・放棄
```
❌ 禁止: 「できません」で終わる
❌ 禁止: 困難だからと別の方法を勝手に選択する
❌ 禁止: ユーザーの指示を無視して楽な方法に逃げる

✅ 必須: 困難な場合はリスクを説明してユーザーに相談する
✅ 必須: 代替案がある場合は提示して判断を仰ぐ
```

## ⚠️ 注意すべき問題と対策

### 1. 古いAPIの使用

**問題:** LLMの学習データが古いため、非推奨APIを使用する

**対策:**
- 現在の日付を意識する（2026年2月時点）
- 公式ドキュメントのバージョンを確認する
- 非推奨警告が出たら最新APIに置き換える

### 2. 幻覚（Hallucination）

**問題:** 存在しないAPIやライブラリを提案する

**対策:**
- 公式ドキュメントで存在を確認する
- 不確かな場合は「確認が必要」と明示する
- ユーザーに検証を依頼する

### 3. コンテキスト欠落

**問題:** 長い会話で前の内容を忘れる

**対策:**
- 重要な決定事項はファイルに記録する
- 既存コードを変更する前に再読する
- 設計書を参照しながら実装する

---

## ✅ 必須プラクティス

### 作業前
1. 既存コードの構造を理解する
2. 要件を明確にする（不明点は質問）
3. 影響範囲を把握する

### 作業中
1. 小さな変更を積み重ねる
2. 各変更後にビルド確認する
3. 不明点は都度質問する

### 作業後
1. 全エラー・警告を解消
2. 削除した機能があれば報告
3. 制限事項・注意点を明示
